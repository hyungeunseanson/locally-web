# Email Notification Policy

## 🎯 목표

이 문서는 `Locally`의 이메일 알림을 언제 보내고, 언제 보내지 않을지 정리한 운영 기준이다.

핵심 원칙은 3개다.

- ✅ 정말 중요한 일만 즉시 메일
- 🔔 나머지는 인앱 알림 우선
- 🧾 팀스페이스는 `10분 digest`로 묶어서 발송

---

## 📌 최종 권장 정책

### 용어 정리

- 🙋 `의뢰자` = 맞춤 서비스를 신청한 고객
- 🧑‍🏫 `호스트` = 그 의뢰에 지원하거나 선택되는 현지 호스트
- 💬 `문의 메시지` = 게스트와 호스트가 주고받는 1:1 메시지
- 🛎️ `서비스` = 체험 예약이 아니라 `맞춤 서비스 의뢰` 흐름

---

### 1. 즉시 이메일 발송

아래는 메일이 바로 가는 것이 맞다.

- 💳 예약 결제 완료
- ❌ 예약 취소 완료
- 🛠️ 호스트 신청 결과
  - 승인
  - 보완 요청
  - 거절
- 📥 무통장 입금 확인 완료
- 📞 프록시 예약 댓글/답변

이유:

- 사용자가 놓치면 실제 운영 문제가 생김
- 돈, 승인, 취소, 상태 변경과 직접 연결됨

---

### 2. 인앱 알림만

아래는 메일 없이 인앱 알림만 권장한다.

- ⭐ 새 후기 등록
- ✍️ 후기 작성 요청
- 🧭 서비스 매칭 일반 상태 변경
  - 새 지원자 등록 → 의뢰자에게 인앱
  - 다른 호스트 선택됨 → 미선택 호스트에게 인앱
  - 서비스 취소 요청 접수 → 상대방에게 인앱

이유:

- 중요하긴 하지만 즉시 메일이 꼭 필요하진 않음
- 자주 발생할 수 있어 메일 피로도가 높음

---

### 3. 10분 Digest 이메일

아래는 즉시 메일 대신 `10분 단위 묶음 메일`이 맞다.

- 👥 팀스페이스 할 일 댓글
- 📝 팀 메모 등록
- 💬 팀 메모 답글
- ✅ 새 할 일 등록
- 🗨️ Team Chat 메시지

권장 방식:

- 같은 사람에게 10분 안에 여러 팀 알림이 생기면 메일 1통만 보냄
- 메일 안에는 요약만 보여줌
- 예:
  - 새 팀 채팅 4건
  - 새 TODO 1건
  - 메모 답글 2건

중요:

- `현재 화면 보고 있던 사람은 메일 생략` 규칙은 넣지 않음
- 단순하게 `10분 단위 digest`만 적용

---

## 🚨 현재 코드 기준 문제점

### 너무 많이 가는 쪽

- 💬 문의 메시지: 메시지 1개마다 바로 메일
- 👥 팀스페이스: 댓글/메모/채팅마다 바로 메일
- 🔁 예약 일부는 중복 알림 가능성 있음

### 꼭 필요한데 안 가는 쪽

- 🛠️ 호스트 신청 승인/보완/거절 메일 없음
- 🏦 무통장 입금 확인 완료 메일 없음
- 🧭 서비스 매칭 주요 상태 메일 정책이 들쭉날쭉함

### 정책이 섞여 있는 쪽

- 어떤 것은 `인앱 + 이메일`
- 어떤 것은 `인앱만`
- 어떤 것은 `이메일만`

즉, 지금은 “기능별 개별 구현” 상태이고 “운영 정책”이 통일되어 있지 않다.

## 🧭 이벤트별 권장 표

| 이벤트 | 인앱 | 이메일 | 권장 방식 |
| --- | --- | --- | --- |
| 예약 결제 완료 | O | O | 즉시 |
| 예약 취소 완료 | O | O | 즉시 |
| 무통장 입금 확인 완료 | O | O | 즉시 |
| 호스트 신청 승인 | O | O | 즉시 |
| 호스트 신청 보완 요청 | O | O | 즉시 |
| 호스트 신청 거절 | O | O | 즉시 |
| 문의 새 메시지(게스트↔호스트) | O | O | 미읽음 10분 후 1회 |
| 후기 작성 요청 | O | X | 인앱만 |
| 새 후기 등록 | O | X | 인앱만 |
| 후기 답글 | O | X | 인앱만 권장 |
| 서비스 새 지원자 | O | X | 의뢰자 인앱만 |
| 서비스 호스트 선택 | O | X | 선택된 호스트 인앱만 |
| 서비스 미선택 | O | X | 미선택 호스트 인앱만 |
| 서비스 결제 확인 | O | X | 의뢰자 인앱만 |
| 서비스 취소 요청/취소 | O | X | 상대방/양측 인앱만 |
| 팀 할 일/댓글/메모/팀채팅 | O | O | 10분 digest |

`문의 새 메시지`는 이제 이렇게 정리하는 것이 좋다.

- 메시지가 오면 일단 인앱 알림
- 상대방이 `10분 안에 읽으면` 이메일 안 보냄
- `10분 동안 안 읽으면` 이메일 1회 발송
- 그 10분 안에 메시지가 3개 와도 메일은 1통만 발송

즉, 문의는 `즉시 메일`보다 `미읽음 10분 후 1회 메일`이 맞다.

## 🛡️ 이메일 인증 권장 정책

### 이메일/비밀번호 가입

- ✅ 이메일 인증 필수

이유:

- 오타 이메일 가입 방지
- 스팸 계정 방지
- 비밀번호 재설정/중요 알림 수신 안정화

### Google / Kakao 가입

- ✅ 별도 이메일 인증 불필요

이유:

- 이미 외부 OAuth 공급자에서 인증된 계정
- 중복 인증을 또 받으면 가입 마찰만 커짐

### 추가 권장

- 📩 `인증 메일 재전송` 버튼 추가
- ⏳ 인증 전 계정은 로그인 제한 유지

## 🏗️ 앞으로의 구현 계획

### Phase 1. 정책 통일

- `이벤트 -> 발송 방식` 매핑표를 코드 기준 단일 소스로 만든다
- 기준값:
  - `immediate`
  - `in_app_only`
  - `digest_10m`

이 단계 목표:

- 기능마다 제각각 메일 보내는 구조를 먼저 멈춤

---

### Phase 2. 팀스페이스 10분 Digest

새 팀스페이스 이벤트는 즉시 메일 대신 큐에 적재한다.

필요한 동작:

- 팀 이벤트 발생
- 이메일을 바로 보내지 않음
- `team_email_digest_queue` 같은 저장소에 쌓음
- 10분마다 cron/worker가 묶어서 발송

한 메일 예시:

- 제목: `[Locally Admin] 지난 10분간 팀 업데이트 7건`
- 본문:
  - 새 채팅 4건
  - 새 TODO 1건
  - 메모 답글 2건

핵심:

- 수신자별 1통
- 10분 창 안의 이벤트 전부 합산
- 즉시 발송 없음

---

### Phase 3. 예약/승인 메일 보강

누락된 중요한 메일을 채운다.

- 🏦 무통장 입금 확인 완료 메일
- 🛠️ 호스트 신청 승인/보완/거절 메일

이건 digest 대상이 아니라 즉시 발송이다.

---

### Phase 4. 문의 메일 재정의

문의는 현재 너무 자주 메일이 간다.

권장 변경:

- 기본은 인앱 알림
- 상대방이 안 읽었을 때만 `미읽음 10분 후 1회` 메일 발송

즉:

- 메시지 올 때마다 메일 X
- 10분 안에 읽으면 메일 X
- 10분 동안 미읽음이면 메일 O
- 같은 10분 창 안 여러 메시지는 메일 1통으로 합침

---

## ✅ 추천 결론

지금 가장 먼저 손대야 할 것은 3개다.

1. 👥 팀스페이스 즉시 메일 중단 후 `10분 digest` 전환
2. 🛠️ 호스트 승인/보완/거절 메일 추가
3. 🏦 무통장 입금 확인 완료 메일 추가

그 다음 단계로:

4. 💬 문의 메시지 메일을 `즉시`에서 `지연/요약형`으로 바꾸는 것이 좋다

`돈/승인/취소는 즉시 메일`, `일반 활동은 인앱`, `팀스페이스는 10분 digest`가 가장 현실적이다.
