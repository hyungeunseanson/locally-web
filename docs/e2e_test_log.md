### 2. 이슈 및 해결
* **이슈**: 신규 가입 시 생년월일(birth_date) 포맷이 유효하지 않을 경우, DB 가입 트리거가 거부되어 유저 생성 자체가 500 에러를 뱉음. 트리거의 EXCEPTION Fallback 블럭이 작동했으나 이 경우 연락처/국적 등을 통째로 버리고 백업 가입만 수행하는 버그 확인.
* **해결**: `LoginModal.tsx` 프론트엔드 제출 폼에 생년월일 정규표현식 유효성 검사 및 연락처 길이 검사 추가. 또한 Sql 트리거 함수(`supabase_profile_sync_v3_migration.sql`)의 예외처리 블록에서도 생일값을 제외한 연락처 등 정상 메타데이터는 누락되지 않도록 Insert문 보강 완료. 
* **이슈**: 신규 가입 이벤트 발생 시 관리자 페이지에 유저 리스트가 즉각 새로고침되지 않음.
* **해결**: `useAdminData.ts`에 `profiles` 테이블의 `INSERT` 이벤트를 감지하는 Supabase Realtime 채널을 활성화하고 토스트 메시지와 함께 `setState` 배열 맨 앞단에 새 유저를 주입함.
* **이슈**: 관리자 대시보드(Users 등) 우측 프로필 카드에서 전화번호/국적/생년월일이 빈 값일 경우 가짜 하드코딩 데이터를 노출하는 UI 버그 확인.
* **해결**: 프론트엔드 단의 하드코딩된 더미 텍스트를 `미입력`으로 변경 및 필드명(`birth_date` 오타 등) 정상 매핑.
* **이슈**: 관리자 대시보드 실시간 접속 유저 명단에 가입된 회원이 비회원으로 뜨고, 클릭해도 아무 일도 일어나지 않는 불편함.
* **해결**: `UserPresenceTracker.tsx`의 실시간 채널(`online_users`) Payload 구성을 변경하여 `full_name`과 `avatar_url`을 백업 조회 후 전송하도록 강화. 또한 인증 상태 변화(`onAuthStateChange`)를 감지해 동기화되게 하고, UI 상에서는 비회원을 가림. 마지막으로 접속 유저 칩을 클릭 시 메인 테이블의 유저 정보 매핑을 통해 상세 프로필 패널이 열리도록 인터랙션 활성화.
* **이슈**: 실시간 접속 유저 명단에서 프로필 사진이 없는 회원의 경우 텍스트명(예: '김')이 표시되어, 기본 리스트의 빈 유저 아이콘(`User`)과 다른 이질감.
* **해결**: `UsersTab.tsx`에서 아바타 fallback 렌더링을 텍스트 첫 글자에서 `lucide-react`의 `<User />` 아이콘 컴포넌트로 변경. 
* **이슈**: 계정 관리에서 사용자 이름을 변경했을 때, 프론트엔드의 트래커가 채널 재생성 시점 차이 등으로 인해 예전 이름을 송신하여 관리자 접속 명단과 실제 리스트의 이름이 불일치.
* **해결**: 첫째, `UserPresenceTracker`에서 매번 채널을 껐다 켜는 대신 기존 채널 구조를 재활용하여 `channel.track(payload)`만 다시 덮어씌움. 둘째, 가장 최신 상태인 관리자 앱의 `users` 테이블 객체(DB State)를 우선 순위로 렌더링하도록 `UsersTab`의 `displayName` 매핑 우선순위 로직 수정 (`const displayName = matchedUser?.full_name || u.full_name`).

### 4. QA 02 (호스트 지원서 및 체험 노출 버킷 관리)
* **이슈**: 관리자 대시보드 호스트 지원 정보에서 '신분증 사진' 썸네일 이미지가 깨지는 현상.
* **해결**: 호스트 지원서 제출 시(`app/host/register/page.tsx`) 공개 `images` 버킷이 아닌 보안 권한이 필요한 `verification-docs` 버킷에 곧바로 업로드하도록 수정. 또한 `DetailsPanel.tsx`에서 이전에 `images` 버킷에 올라갔던 공개 파일들을 볼 수 있도록 Fallback URL 렌더링 로직 추가.
* **이슈**: 호스트가 정식 승인(`approved`)되기 전의 대기 상태(`pending`)이거나 재심사(`revision`) 상태임에도 불구하고, 호스트가 임의로 먼저 작성해둔 '활성 체험'이 메인 및 검색 페이지(사용자 목록)에 노출되고 예약 가능한 상태가 됨.
* **해결**: 메인 화면/검색에 사용되는 API(`app/utils/api/experiences.ts`) 및 유저 프로필 조회 페이지(`users/[id]/page.tsx`)에서, 체험(`experiences`)을 조회하기 전 `host_applications` 테이블의 `status`를 선제적으로 조회하여 반드시 `'approved'`인 호스트의 체험만 필터링하여 데이터를 병합해 응답하도록 쿼리 조인 가드 로직 적용.

_새로운 E2E 테스트 프로세스나 QA 내용이 있을 경우, 이 문서의 상단이나 하단에 계속해서 누적 기록합니다._

### 5. QA 03 (관리자 대시보드 체험 호스트 식별명 추가)

### 6. QA 04 (관리자 대시보드 팀 워크스페이스 실시간 동기화 버그)
* **이슈**: 팀 워크스페이스(`TeamTab.tsx`)에서 채팅 메시지를 전송하거나 미니 채팅(`MiniChatBar`)을 사용할 경우, 화면에 보이는 모든 '할 일(Todo)'의 댓글이 순식간에 사라졌다가 새로고침 시 다시 복구되는 현상 발생.
* **원인**: 수신 범위가 넓은 실시간 구독(Realtime Subscription) 리스너가 채팅 메시지 INSERT 이벤트를 감지할 때, `TeamTab.tsx`의 텅 빈 의존성 배열(`[]`)로 인해 초기 렌더링 시점의 빈 상태(`tasks = []`)를 참조하는 Stale Closure(과거 상태 참조) 버그가 발동함. 이로 인해 `fetchComments()`가 필터링할 ID를 찾지 못하고 댓글 상태를 빈 배열(`[]`)로 덮어씌워버림.
* **해결**: `useRef`를 사용하여 가장 최신의 `tasks` 리스트(`tasksRef.current`)를 유지하도록 수정함. 이를 통해 `fetchComments` 함수 안심하고 실시간 콜백에서도 현재 화면에 띄워진 할 일 ID 목록을 온전히 가져와 댓글을 필터링 및 유지할 수 있게 됨.

### 7. QA 05 (관리자 대시보드 체험 승인 탭 수정 오류 및 사진 노출 버그)
* **이슈**: 관리자 대시보드 체험 승인 탭에서 새롭게 등록된 투어의 사진이 빈칸으로 나오며, '관리자 권한으로 수정하기' 클릭 시 "데이터를 불러오지 못했습니다" 오류 후 대시보드로 튕기는 현상 발생.
* **원인**:
  1. (수정 튕김): `EditExperiencePage` 상단과 데이터 저장 로직에서 `host_id`가 현재 로그인한 유저(`user.id`)와 일치해야만 조회/수정이 가능하도록 강력한 RLS 우회 제약조건(`.eq('host_id', user.id)`)이 프론트엔드에 하드코딩 되어 있었음. 이로 인해 관리자가 접근 시 권한 부족으로 튕겨남.
  2. (사진 빈칸): 제출된 이미지들이 저장되는 Supabase Storage의 `experiences` 버킷 엑세스 권한이 공개되지 않아, 화면에서 `<img src>` 렌더링 시 권한 오류(`403 Forbidden`)를 뱉음.
* **해결**:
  1. `EditExperiencePage` 상단에 접속한 유저가 관리자(`profiles.role === 'admin'` 또는 `admin_whitelist` 포함 여부)인지 확인하는 예외 처리 로직(`isAdminUser`)을 추가하고, 관리자일 경우엔 `host_id` 필터링 검사를 우회(Bypass)하여 데이터 패치 및 저장(Update)이 모두 가능하도록 코드를 수정함.
  2. 사진 빈칸 문제는 코드 수정이 아닌, 운영 환경에서 `experiences` 버킷을 'Public'으로 퍼블릭 액세스 설정을 부여하는 것으로 보고 및 가이드 조치 (코드 수정 X).

### 8. QA 06 (홈 화면 및 프로필 페이지 활성 체험 미노출 버그)
* **이슈**: 호스트가 정식 승인(`approved`)되고 체험이 `active` 상태임에도 불구하고, 메인 홈 화면 및 유저 프로필 페이지에서 활성 체험이 노출되지 않는 현상 발생.
* **원인**: 최근 추가된 `host_applications` 테이블의 RLS(Row Level Security) 정책(`ha_select_own`)으로 인해, 비로그인 일반 게스트가 승인된 해당 호스트의 지원서 상태를 조회할 시 권한 부족으로 빈 배열(`[]`) 또는 `null`이 반환되면서 체험 렌더링 로직이 모두 차단됨.
* **해결**: 개인정보(계좌번호, 전화번호 등)를 제외하고 안전한 읽기 권한(`anon`, `authenticated`)이 부여된 데이터베이스 뷰(Postgres View) `public_host_applications`를 DB에 생성. 이후 프론트엔드 관련 파일 3곳(`app/utils/api/experiences.ts`, `app/users/[id]/page.tsx`, `app/experiences/[id]/page.tsx`)의 쿼리를 해당 뷰를 조회하도록 모두 수정함.
