# Locally E2E 통합 테스트(Epic 시나리오) 실행 기록

본 문서는 `tests/e2e/01-fully-integrated-ecosystem.spec.ts` 스크립트를 통한 통합 E2E 봇 탐사의 시도 내역과 발생한 에러, 해결 과정을 기록합니다.

## 시도 1: 회원가입 모달 모드 전환 버튼 인식 에러
- **진행 상황**: 시작하자마자 실패
- **문제 원인**: 호스트 가입 시 사용하는 Login 모달의 하단 텍스트 "이미 계정이 있으신가요? 계정 생성" 버튼을 `text=SIGNUP` 등의 모호한 텍스트 셀렉터로 잡아 클릭하려다 타임아웃 발생.
- **해결 방안**: DOM 구조 `div.mt-6.text-center.text-sm > button`를 직접 참조하도록 CSS 기반 셀렉터로 변경함.

## 시도 2: 관리자 대시보드 호스트 지원 승인 버튼 인식 에러
- **진행 상황**: Scene 1 (호스트 가입) 완료 후 관리자 로그인 진입 성공했으나 승인 탭에서 실패
- **문제 원인**: 코드상에는 사용자 메뉴(Users)로 가서 tr>td 구조의 표에서 이메일을 찾는 로직으로 되어 있었으나, 실제 관리자 화면(Sidebar)에는 'Approvals' 탭에서 ListPanel UI 렌더링을 사용함.
- **해결 방안**: 관리자 화면 컴포넌트(`Sidebar.tsx`, `ManagementTab.tsx`, `ListPanel.tsx`, `DetailsPanel.tsx`)를 분석하여 리스트 아이템 클릭, "승인 (호스트 권한 부여)" 텍스트 버튼 클릭 등 실제 UI 구조에 맞게 코드 전면 개편.

## 시도 3: 빈 호스트 지원 폼 즉시 제출 에러
- **진행 상황**: 시작 시 Become a Host 페이지(히어로 섹션)에서 호스트 등록하기 버튼 누른 직후失败
- **문제 원인**: 기획상 8단계에 이르는 호스트 지원 폼(국적, 언어, 연락처 등)이 구현되어 있었으나 테스트 봇 코드가 이 폼을 1단계만 지나고 넘기려고 하면서 타임아웃 발생. 
- **해결 방안**: 8단계 가입 폼의 각 렌더링 요소들을 하나씩 클릭하고 채우는 UI 자동화 액션을 작성하여 제출 완료까지 구현함.

## 시도 4: 회원가입 세션 생성 전 라우팅 에러
- **진행 상황**: Scene 1 가입 8단계 완료 직전 또는 직후 에러 발생
- **문제 원인**: LoginModal 내 회원가입 API 로직(Supabase Auth) 완료 전 비동기적으로 `goto('/become-a-host')` 라우팅이 실행되어 로그인 세션 유지 실패.
- **해결 방안**: 가입 폼 제출 후 완료 토스트(가입 처리)가 DOM에 보일 때까지 대기(waitForTimeout)하는 코드 주입.

## 시도 5: Database (Supabase) 스키마 변경 캐시 에러
- **진행 상황**: 가입 완료(Supabase Auth) 요청 시 400 Bad Request 에러
- **문제 원인**: Playwright 브라우저 콘솔을 열어 분석한 결과 `{code: PGRST204, message: Could not find the 'language_levels' column of 'host_applications' in the schema cache}` 가 발생함.
- **해결 방안**: 데이터베이스에서 SQL 마이그레이션을 통해 `language_levels` 컬럼이 추가되었으나 PostgREST 캐시가 업데이트되지 않아 생긴 에러. 관리자(대표님)를 통해 `NOTIFY pgrst, 'reload schema';` 명령어 실행 완료.

## 시도 6: 스키마 재적용 후 전체 시나리오 재실행
- **진행 상황**: Scene 1 폼 전송버튼 클릭 직후 타임아웃
- **문제 원인**: 사용자(대표님)께서 `NOTIFY pgrst, 'reload schema';` 캐시 새로고침 명령어를 정상 실행해주셨음에도 동일한 에러(`PGRST204: Could not find the 'language_levels' column`)가 재차 발생함.
- **분석 결과**: 이는 캐시 새로고침 실패가 아니라 애초에 DB의 `host_applications` 테이블에 `language_levels` 컬럼이 생성되지 않았을 확률이 99%입니다. 즉 `supabase_language_levels_migration.sql` 전체 파일 내용이 한 번도 실행되지 않았을 가능성이 대두됨.
## 시도 7: 사용자의 피드백 제기 및 원인 구명
- **상황**: 사용자(대표님)께서 "기존 어드민 화면에서 체험 상세에 진행 언어 Lv.n이 잘 나오는데 왜 컬럼이 없다고 하느냐?"며 이의를 제기하셨음.
- **분석 결과**: 
  1. 직접 Node.js 기반 Fetch Script로 Supabase API에 `language_levels` 컬럼 조회를 강제 요청한 결과, Postgres 42703 (`undefined_column`) 에러 코드가 똑똑히 반환됨. 즉 DB에는 두 테이블 모두 해당 컬럼이 **없음**.
  2. 프론트엔드 코드를 분석(`grep_search`)한 결과, `ExpMainContent.tsx`와 `DetailsPanel.tsx` 등에서 `normalizeLanguageLevels()` 함수를 호출하는데, 이 함수는 `language_levels` 컬럼이 비어있으면 **구형 `languages` 컬럼(단순 배열)을 파싱하여 강제로 화면에만 Lv.3으로 표시하는 Fallback 로직**이었음. (즉, 착시 현상)
- **다음 조치**: 사용자에게 직접 마이그레이션 파일 내용을 한 번 더 실행해 주실 것을 강력히 권고한 후 대기.

## 시도 8: 진짜 스키마 적용 후 대망의 재실행
- **진행 상황**: 대표님께서 전체 컬럼 생성 및 데이터 매핑 마이그레이션 SQL 구문을 실행 완료하셨음.
- **예상 결과**: `language_levels` 컬럼 부재로 발생하던 400 Bad Request 에러가 해소되고, 8단계 지원 폼 제출이 성공하여 관리자 대시보드(승인 뷰)로 매끄럽게 넘어갈 것임.
- **결과**: **[성공]** 처음으로 [Scene 2] 진입 로그를 띄우고 호스트 권한 획득 성공! 하지만 Scene 2의 체험 생성 코드 역시 8단계 폼 진입으로 되어 있지 않아 타임아웃 발생.

## 시도 9: 체험 생성 8단계 위저드 스크립트 재작성 후 재실행
- **진행 상황**: 단일 간소화 폼이던 Scene 2 (체험 생성) 코드를 1~8단계 위저드를 순차적으로 클릭하고 이미지를 업로드하는 실제 유저 흐름으로 전면 개편함.
- **결과**: **[실패]** 8단계 중 Step 4(주소)와 Step 6(연령)의 입력란 Placeholder 텍스트를 잘못 타겟팅하여 타임아웃 발생 (실제 앱의 Placeholder 텍스트와 스크립트의 타겟팅 불일치).

## 시도 10: Placeholder 셀렉터 픽스 후 재실행
- **진행 상황**: `input[placeholder*="구글맵에서 검색 가능한"]` 등을 `input[placeholder*="예) 서울특별시"]` 등 실제 렌더링 값으로 올바르게 수정함.
- **결과**: **[실패]** 어이없게도 처음 시작점인 Scene 1의 호스트 회원가입 부분에서 타임아웃(`/become-a-host` 진입 후 `/host/register`로 전환 안 됨). 조사 결과 회원가입 완료 토스트를 대기하는 정규식(`/가입|완료|환영합니다/i`)이 토스트가 뜨기도 전에 폼의 **"가입하기" 버튼**에 먼저 매칭되어 곧바로 다음 단계로 넘어갔고, 이 탓에 회원가입 API 통신이 강제 중단(`Failed to fetch`)되어 세션이 맺어지지 않은 상태에서 계속 헛돌고 있었음.

## 시도 11: 회원가입 셀렉터 버그 픽스 후 재실행
- **진행 상황**: 완료 토스트 검증 정규식을 엄격하게(`/가입 완료|성공적으로|환영합니다/i`) 변경하여 "가입하기" 버튼과 오작동을 차단함.
- **결과**: **[실패]** 이메일 인증 기능이 꺼진 현재 DB 환경에서는 `LoginModal.tsx` 로직상 가입 성공 시 **어떤 토스트 알림도 아예 띄우지 않고 창만 조용히 닫아버림**이 뒤늦게 판명됨. 결국 나타나지 않는 토스트를 15초간 기다리다가 또 타임아웃 발생.

## 시도 12: 궁극의 모달 닫힘 감지 & API 통신 대기 병합 후 실행
- **진행 상황**: 불안정한 UI 텍스트 검증을 영구 폐기하고, 브라우저가 Supabase 측에 쏘는 `auth/v1/signup` API 요청의 200 OK 응답 패킷을 네트워크 단에서 가로채 대기한 뒤, 물리적인 모달창 파괴(`not.toBeVisible`)를 교차 검증하는 스크립트로 진화시킴.
- **결과**: **[실패]** Scene 2 (체험 생성)의 Step 3에서 사진을 업로드한 직후, 서버가 이미지를 처리하여 미리보기를 띄우기도 전에 봇이 '다음' 버튼을 광클릭함. 이로 인해 사진 누락 validation 에러가 발생하여 Step 4로 넘어가지 못하고 타임아웃 발생.

## 시도 13: 사진 업로드 안정성 보강 후 재실행
- **진행 상황**: Step 3에서 이미지를 `setFiles` 한 후, 그 결과물인 미리보기 `img` 태그가 DOM에 실제로 나타날 때까지 대기하는 로직을 추가함.
- **결과**: **[실패]** 드디어 Step 7(요금 설정)까지 마침내 뚫어냈으나, 최종 화면에서 "체험 등록 완료!"라는 텍스트를 찾으려다 타임아웃 발생. 확인 결과 실제 코드는 등록 성공 시 Success 화면(Step 8)을 보여주기보다 관리자 대시보드로 즉시 리다이렉트(`router.push`)해버리고 있었음.

## 시도 14: 로그 강화 및 리다이렉트 검증 기반 재실행
- **진행 상황**: 각 단계마다 `console.log`를 출력하도록 하여 어느 단계에서 막히는지 가독성을 높였으며, 최종 성공 조건을 "Step 8 텍스트 대기"가 아닌 "Dashboard URL 리다이렉트 대기"로 수정함.
- **결과**: **[실패]** 리다이렉트는 성공했으나, 대시보드 로딩 속도 문제 혹은 '내 체험' 텍스트의 미묘한 로케이터 매칭 실패로 타임아웃 발생. (실제 UI 텍스트는 '내 체험 관리'임)

## 시도 15: 정규식 로케이터 및 네트워크 대기 보강 후 재실행
- **진행 상황**: `text=내 체험` 대신 `/내 체험|My Exp/i` 정규식을 사용하고, `networkidle` 대기를 추가하여 대시보드 데이터 렌더링 후 검증하도록 개선함.
- **예상 결과**: Scene 2의 최종 관문을 넘고, 승인 처리(Admin) 후 게스트 예약(Scene 3)으로 부드럽게 이어질 것임.
